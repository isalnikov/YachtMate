# Фаза 7 — Безопасность, журнал, экипаж, документы

**Цель:** электронный судовой журнал; SOS с отправкой координат доверенным контактам и каналам по региону; запись треков; чек-листы выхода и швартовки; хранение документов (страховки, лицензии); роли экипажа и синхронизация общих данных офлайн-first.

**Зависимости:** GPS и позиционирование из [Фазы 1](phase-01-navigation-core.md); при необходимости SMS/email — платформенные API.

**Ссылки:** [`../TODO.md`](../TODO.md), **[`../Подробный план реализации.md`](../Подробный%20план%20реализации.md)** (модуль M10, SOS severity `critical`, роли §5.5, Приложение A).

---

## Связь с финальным планом

- **TDD:** тесты репозиториев журнала, документов и треков; сценарий SOS офлайн → очередь синка; экспорт данных.
- **Аудит:** `logbook_entry_create` и связанные действия; SOS — **`critical`** в телеметрии/аудите по политике продукта; завершение чек-листа безопасности — отдельное событие (`safety_checklist_complete`).
- **Логирование:** координаты и персональные данные не попадают в открытые логи; маскирование согласно §10 финального плана.

## Шаги и артефакты

| Шаг | Действие | Выход |
|-----|----------|--------|
| 7.1 | Журнал: записи событий, расход топлива, техобслуживание; экспорт (CSV/PDF опционально) | Локальная БД — **см. `LogbookEntries`, вкладка «Ещё», экспорт CSV в буфер** |
| 7.2 | SOS: подтверждение действием (long press / два шага); координаты, время, судно; SMS/email/API | **`lib/features/distress/`**, тестовый режим + аудит `critical`, без координат в `context_json` |
| 7.3 | Трек рейса: запись точек по интервалу или по изменению курса; история на карте | **`TrackTrips`/`TrackPoints`**, таймер 30 с + линия на карте |
| 7.4 | Чек-листы: шаблоны выхода в море, швартовки, штормовой готовности | **3 шаблона**, аудит `safety_checklist_complete` |
| 7.5 | Документы: шифрование на устройстве (Keychain/Keystore); опционально облако E2E | **AES + PIN**, метаданные в SQLite, `file_picker` |
| 7.6 | Роли экипажа (капитан/матрос): права на действия; синхронизация через conflict-free структуры где возможно | **Инвайт-код → ship_id (хэш)**; удаление журнала/сейфа — только капитан (локально) |

## Детализация

### 7.2 SOS

- Тестовый режим без реальной отправки.
- Региональные номера спасательных служб — конфигурируемо, не хардкод без проверки.

### 7.6 Синхронизация экипажа

- Ship как «пространство» с invite-кодом; аудит действий для безопасности.

---

## Критерии приёмки

- Журнал переживает перезапуск приложения и резервное копирование ОС где применимо.
- SOS в тестовом режиме не отправляет внешние сообщения без явной метки «TEST».
- Документы недоступны без разблокировки приложения (если включено шифрование).

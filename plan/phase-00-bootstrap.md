# Фаза 0 — Инициация продукта и инфраструктура

**Цель:** единые правила разработки, рабочий репозиторий, CI, качество кода, дизайн-система и основа локализации.

**Зависимости:** нет (старт проекта).

**Ссылки:** [`../TODO.md`](../TODO.md), [`README.md`](README.md), **[`../Подробный план реализации.md`](../Подробный%20план%20реализации.md)** (TDD §9, логирование §10, дерево пакетов §3).

---

## Связь с финальным планом реализации

- **TDD:** все последующие фазы подчиняются циклу Red → Green → Refactor; порог покрытия `lib/domain/**` и `lib/data/repositories/**` **≥90%** (исключения только в списке `coverage_exclusions.yaml`).
- **Логирование:** до выхода на любой экран после дисклеймера должны существовать **`AppLogger`** (технический канал) и **`AuditRepository`** + таблица Drift **`user_action_audit`** (см. §5.2 и §10 финального плана).
- **Структура каталогов:** ориентир — `lib/core/logging`, `lib/domain`, `lib/data`, `lib/features/*`, `lib/hardware` (см. §3 финального плана).

## Детализация подшагов

### 0.1 Scope MVP

- Перечислить обязательные экраны и отказ от функций следующих фаз.
- Согласовать с [`phase-01-navigation-core.md`](phase-01-navigation-core.md): что входит в первый релиз магазина.

### 0.2 ADR по фреймворку

- Документировать альтернативу (например KMP только Android) и причину отказа.

### 0.3 Репозиторий

- Структура папок под слои (presentation / domain / data / hardware) или feature-first — одно правило на весь проект.

### 0.4 CI

- Минимум: `flutter test --coverage` + **fail job** при падении ниже порога для `domain`/`data` (скрипт поверх `lcov`).
- Сборка debug APK; при наличии macOS runner — iOS build.
- Документировать в README команду локальной проверки покрытия.

### 0.3a Логирование и аудит (обязательно до Фазы 1 UI)

- Реализовать запись в `user_action_audit` для тестового события при старте (проверка канала).
- Технические логи не содержат токенов API и точных координат в release-сборке (см. финальный план §10.3).

### 0.5 Fastlane

- Отдельные lane для загрузки в Internal Testing / TestFlight при наличии секретов.

### 0.6 Тема

- Контрастность для солнечной палубы; отдельная палитра night vision при необходимости (можно отложить частично в Фазу 8).

### 0.7 i18n

- Fallback язык (EN); не хардкодить строки в UI.

## Критерии приёмки фазы

- Приложение собирается локально и в CI для целевых платформ.
- На устройстве/эмуляторе запускается shell с применённой темой и сменой языка RU ↔ EN без падений.
- Документ scope MVP и ADR по стеку доступны в репозитории.
- **`AppLogger` и `AuditRepository` работают;** таблица аудита создаётся миграцией; есть минимальный тест записи аудита.
- CI выполняет тесты с **coverage** и **анализ** без ошибок.
